# WaterLevelMonitor Test Strategy

本ドキュメントでは、WaterLevelMonitor プロジェクトにおけるテスト戦略（何を単体テストし、何をE2E/統合テストでカバーするのか）をモジュールごとに定義します。

## 1. テスト方針の基本原則
*   **Unit Test (単体テスト)**: 外部ネットワーク (HTTP) やデータベース (Supabase) に依存しない、副作用のない「純粋なロジック」や「データ変換処理」を対象とします。境界値やエッジケースの検証を高速に行うことを目的とします。
*   **E2E / Integration Test (統合テスト)**: モジュール間の連携、外部API（ダミー）の呼び出しからDB（モック）への保存まで、一連のパイプライン全体が正しく動作することをDBやHTTP通信を使わずにモックとテストデータを使うことで確認します。

---

## 2. モジュールごとのテストスコープ

### `src/scraper.py`
*   **役割**: 指定URLからHTMLを取得し、DATファイルのリンクを解析してPandas DataFrameとして抽出する (Extract)。
*   **Unit Test**: 
    *   **対象**: HTTP通信をモック化（`responses` や `pytest-mock` を使用）し、ローカルの `fixtures/` にある生データ（HTMLやDAT）を注入してパースロジックをテストする。
    *   **検証項目**: DataFrameの列数、ヘッダー名の上書き処理、不要な改行やスペースのトリムが正しく行われているか。欠測値 (`$`) や未受信 (`-`) の行が文字列として正しくDataFrameに格納されているか（ここではまだ加工しない）。

### `src/converter.py`
*   **役割**: 生のDataFrameをDB向けの辞書リストに変換する (Transform)。
*   **Unit Test**:
    *   **対象**: 最も重要なテスト対象。ローカルのダミーDataFrameを入力し、変換結果を検証する。DBアクセス（`_get_latest_timestamp`, `_save_to_db`）はモック化して分離する。
    *   **検証項目**: 
        *   `_safe_float`: `None`, `"-"`, `"$"`, 文字列, 正常値の変換精度。
        *   タイムゾーンの変換処理（JSTからUTCへの変換、`24:00` の翌日`00:00` 繰り上げ処理）。
        *   カラム名のマッピング（ダム向けと雨量向けの違い）。
        *   最新タイムスタンプ (`latest_ts`) を基準とした差分データの正確な抽出。

### `src/plot.py`
*   **役割**: DataFrameからMatplotlibのFigureオブジェクトを生成する (Visualize)。
*   **Unit Test**:
    *   **対象**: 画像の視覚的検証は行わず、データの集約や描画設定の計算ロジックをテストする。
    *   **検証項目**: 
        *   処理の途中でエラー（クラッシュ）が発生せず `Figure` が返却されるか。
        *   **データ集約**: 10分雨量のデータが、`Axes` オブジェクト上で1時間単位の積算データに正しくまとめられているか抽出してアサート。
        *   **表示計算**: データセット（最大値・最小値）を与えた際に、Y軸の上限・下限 (`get_ylim()`) が意図したマージンを持って設定されているか。

### `src/storage.py`
*   **役割**: Supabaseとの通信、バッチUPSERT、ページネーション取得 (Load)。
*   **Unit Test**: **対象外**。
    *   **理由**: 処理の95%がSupabaseクライアントライブラリの呼び出しであり、これを単体テストでモック化してもライブラリの使い方のチェックにしかならず、ロジックの安全性担保としての費用対効果が低いため。
*   **Integration / E2E Test**: `test_pipeline.py` における `mock_supabase`（モック環境）、または将来的に導入する「検証用Supabaseプロジェクト」への実際の接続（統合テスト）でカバーする。

### `src/app.py` & `src/main.py`
*   **役割**: Streamlit UIの構築とパイプラインの呼び出し司令塔。
*   **Unit Test**: **対象外**。
    *   **理由**: 状態を持続させるUIフレームワーク（Streamlit）への依存度が高く、また単体で検証すべき複雑なビジネスロジックを持たないため。
*   **E2E Test**: `main.py` の実行フローは現在の `test_pipeline.py`（パイプライン全体の疎通確認と JSON Snapshot による出力一致確認）によってカバーされている。

## 3. 現在のテスト戦略でカバーされていない領域（未テスト領域）
費用対効果や現在のプロジェクトフェーズ（運用初期・個人開発寄り）を考慮し、以下の領域は意図的に**テスト自動化の対象外**としています。

1.  **外部APIの実接続テスト (Live Integration)**
    *   **内容**: 実際の「川の防災情報」サイトが予期せず仕様変更・レイアウト変更を行っていないかを検知するテスト。
    *   **理由**: 外部サイトへの負荷を避けるため。また、定期実行CIに組み込むと相手方のメンテナンス等で意図せずテストが落ちる（Flaky Test）原因になるため。
2.  **Supabase 実DBへの統合テスト (DB Integration)**
    *   **内容**: 本物の PostgreSQL インスタンス（または検証用プロジェクト）を立ち上げ、INSERT/SELECT が1000件のページネーションの境界含めて本当に成功するどうかのテスト。（現在のE2Eテストはモック化された辞書配列への突合に留まっている）
    *   **理由**: 検証用DBインスタンスの用意と自動リセット用のマイグレーションスクリプト等のCI連携コスト（環境構築の手間）が高いため。
3.  **Streamlit UI のブラウザ操作テスト (UI / Visual Regression)**
    *   **内容**: Selenium や Playwright などを使い、「セレクトボックスで別のダムを選んだら即座に再描画されるか」「グラフが画面崩れなく表示されるか」をブラウザ上でシミュレートするテスト。
    *   **理由**: 今回はインフラ・データ層の堅牢化が主目的であり、UIレイヤーの自動テストは保守コストが高すぎるため。目視確認（手動検証）に留める。
