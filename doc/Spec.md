# 水位監視アプリケーション仕様書 (Spec.md)

## 概要
川の防災情報（国土交通省）の特定ダム・観測所の水位データを取得し、グラフとしてわかりやすく可視化するアプリケーション。

アーキテクチャ・技術選定・ADR は [Architecture.md](Architecture.md) を参照。

---

## 1. 目的（Goal）
- 指定URLから特定のダム・観測所の水位・貯水量・雨量・流入出量データを取得する。
  - 今後他のダムや観測所にも容易に対応できるよう、URLの一部（観測所ID等）を外部設定ファイル（`dams.yaml`）から動的に変更・指定できる設計とする。
- 取得したデータをパースし、時系列グラフとして表示することで現在の水位やトレンドを即座に把握できるようにする。

---

## 2. 機能要件 (Functional Requirements)

### 2.1 データ取得・蓄積
- 対象URLから最新のデータを取得する。取得元は DATファイル（CSV互換）を直接ダウンロードし、HTMLパースは行わない。
- 取得元のデータ期間制限（過去1週間分など）を補うため、取得した生データを Supabase（PostgreSQL）に差分UPSERTして蓄積する。
  - DBの最新タイムスタンプ以降のデータのみを抽出・保存（差分UPSERT）。
  - `station_id` + `timestamp` を主キーとし、重複を排除する。
- **タイムゾーンの取り扱い**
  - 取得元の時刻データ（JST）は、Transform時に**UTC**へ変換してDBに保存する。
  - グラフ描画直前に**JST**へ変換して表示する。
- データ収集は GitHub Actions Cron（20分おき）で `src/pipeline.py` が自動実行する。Web UI（`src/app.py`）はスクレイピングを行わず、DBからの読み込みに専念する。

### 2.2 スロットリングガード
- `pipeline.py` / `main.py` から呼ばれる `refresh_data()`（`converter.py`）に実装。
- DBの最新タイムスタンプを確認し、**前回取得から20分以内の場合はスクレイピングをスキップ**する。
- これにより、取得元サーバー（river.go.jp）への過剰なアクセスを防ぐ。

### 2.3 グラフ描画
横軸に時間、縦軸にデータを配置した複合グラフ（サブプロット）を表示する。

- **上段（メイン）**: 貯水率の折れ線グラフと雨量の棒グラフを重ねて表示する。
  - 貯水率は、貯水量データと `dams.yaml` に設定された有効貯水容量から計算する（例: 宮ヶ瀬ダム = 183,000,000 m³）。
  - 雨量データは10分ごとの生データを **1時間ごとの積算雨量（`resample('1h')`）** に集約してから棒グラフ表示する（Y軸ラベル: mm/h）。
- **下段（サブ）**: ダムへの流入量と放流量を棒グラフ（放流量はマイナス方向）で表示し、流量の収支を可視化する。
- ダムの `rain` 設定が省略されている場合（利根川水系等、DATにデータが内包される場合）は雨量グラフを表示しない。

### 2.4 Web UI
- ダム選択（`st.selectbox`）でグラフを切り替えられる。
- 表示期間をラジオボタン（**2週間 / 1年 / 全部**、デフォルト: 2週間）で切り替えられる。DBクエリレベルで期間フィルタをかけるため、大量データでも高速に動作する。
- アプリ末尾に `doc/Disclaimer.md` の内容を免責事項として表示する。

## 3. 懸念事項・留意点 (Risks & Considerations)
- **スクレイピングの安定性**: 対象サイトの仕様変更（URLパラメータ変更等）によってデータが取得できなくなるリスク。
- **Cron停止時のデータ鮮度**: GitHub Actions の Cron が止まる（Secrets 未設定など）と、Web UI が表示するデータが古くなる。`SUPABASE_URL` / `SUPABASE_KEY` の Secrets 設定が正常稼働の前提。
- **`24:00` 問題**: 取得元データに `24:00` 表記が含まれる場合、`converter.py` で翌日 `00:00` に繰り上げ処理を行う。
- **`$` / `-` の取り扱い**: `-`（未受信）の行はフィルタリングして保存しない。`$`（欠測）の値カラムは `None`（DB上は NULL）に変換する。
