# 水位監視アプリケーション仕様書 (Spec.md)

## 概要
川の防災情報（国土交通省）の特定ダム・観測所の水位データを取得し、グラフとしてわかりやすく可視化するアプリケーション。

## 1. 目的（Goal）
- 指定URL（例: `https://www1.river.go.jp/cgi-bin/DspDamData.exe?ID=1368030799020&KIND=3&PAGE=0`）から特定のダム・観測所の水位データを取得する。
  - 今後他のダムや観測所にも容易に対応できるよう、URLの一部（観測所ID等）を外部設定（環境変数や設定ファイル、入力フォーム等）から動的に変更・指定できる設計とする。
- 取得したデータをパースし、時系列グラフとして表示することで現在の水位やトレンドを即座に把握できるようにする。

## 2. 機能要件 (Functional Requirements)
- **データ取得・蓄積機能**
  - 対象のURLから最新のデータ（ページ上の過去履歴）を取得する。
  - 取得元のデータ期間制限（過去1週間分など）を補うため、取得した生データをクラウドデータベース（Supabase / PostgreSQL）に保存・蓄積する。次回以降の取得時は、過去データとの差分（新規データのみ）を追記(UPSERT)していく仕組みとする。
  - **タイムゾーンの取り扱い**
    - 取得元の時刻データ（JST）は、バックエンド処理およびデータベース保存時に自動で**UTC（協定世界時）**に変換し、統一フォーマットで保管する。
    - グラフ描画などのフロントエンド表示の直前で**JST（日本標準時）**に変換して表示し、内部データと表示時刻の整合性を保つ。
  - Webサイト特有のエンコーディング（必要であればShift-JIS等）を適切にデコード・パースして、時間・数値の構造化データ（JSON等）に変換する。
- **グラフ描画機能**
  - 横軸に時間、縦軸にデータを配置した複合グラフ（サブプロット）を表示する。
  - **上段（メイン）**: 水位または**貯水率**の折れ線グラフと、**雨量**の棒グラフを重ねて表示する。
    - 貯水率は、取得した貯水量データと、各ダムの事前設定された満水時データ（有効貯水容量など）を用いて計算する（例: 宮ヶ瀬ダムの有効貯水容量 = 1億8300万 m³）。
    - 雨量データは、10分ごとの生データだと棒グラフが潰れるため、**1時間ごとの積算雨量に集約（リサンプリング）**して表示する。
  - **下段（サブ）**: ダムへの**流入量**と**放流量**を棒グラフ（放流量はマイナス方向）として表示し、流量の収支を可視化する。
  - ツールチップで特定時刻の詳細数値を表示できるインタラクティブなUI。
- **データの自動・手動更新**
  - アクセス時に最新情報を取得するか、ユーザーの操作（更新ボタン等）によって最新データを反映する。

## 3. 非機能要件・アーキテクチャ案 (Architecture & Tech Stack)
- **設定データ管理**
  - ダムごとの設定（観測所ID、有効貯水容量、ダム名など）は別ファイル（`dams.yaml` 等）に外出しして管理する。これにより、将来的に他のダム・観測所への対応がコーディングなしで行えるようにする。
  - Pythonでのパース・型検証には最新トレンドである `Pydantic` などを活用し、設定ファイルの読み込みエラー等を防ぐ。
- **プラットフォーム**: 
  - **初期（Local MVP）**: ローカルPCで動作するPythonスクリプトやNode.jsスクリプトとして開発し、グラフを出力・保存する、またはローカルサーバー（localhost）で閲覧するシンプルな構成。
  - **将来（Webアプリ化）**: 変更を最小限に抑えつつ、Webサービスとして公開可能なアーキテクチャ。
- **技術スタック（低コスト・無料枠運用を前提）**:
  - **[現在採用] 案1: Python (Streamlit)**
    - **採用理由**: **圧倒的に手っ取り早い**から。Pythonのみでバックエンドロジック（データ取得・DB保存）からフロントエンド（グラフ描画UI）まで一貫して作成でき、HTML/CSS等の知識なしでMVP（最小限の実装）を最速で構築・公開（Streamlit Community Cloud等で無料）できるため、初期フェーズの選択肢として最適と判断しました。
    - **デメリット**: リッチなUIデザインの自由度はやや下がる。
  - **[将来の移行候補] 案2: Python (FastAPIロジック) + Vercel (Next.js/React)**
    - **位置付け**: 将来アクセス数が増加した場合や、より高度なフロントエンド（レスポンシブでリッチなUI）を作り込みたくなった場合の移行先。バックエンドはVercelのServerless Functions（Python）に載せれば無料で運用可能。
    - **デメリット**: フロント開発とデータ取得ロジックのそれぞれに学習と設定が必要。
  - **[将来の移行候補] 案3: Node.js フルスタック (Next.js または Hono)**
    - **位置付け**: 展開規模がさらに大きくなった場合、ひとつのプロジェクト（言語）で完結させ、VercelやCloudflare Pages/Workers等、強力なエッジネットワーク・無料枠で手軽かつ高速に動作するWebアプリ化を目指す統合構成。
    - **デメリット**: スクレイピングツール（Cheerio等）の扱いへの適応や、既存のPython資産（グラフ等）からの書き換えコストがかかる。
## 4. 懸念事項・留意点 (Risks & Considerations)
- **スクレイピングの安定性**: 対象サイトの仕様変更（URLパラメータやHTML構造の変更）によってデータが取得できなくなるリスク。
- **サーバー負荷への配慮**: 対象サーバー（river.go.jp）へのアクセス頻度が過剰にならないようにするため、データ取得（スクレイピング）処理の前にデータベース上の最新データのタイムスタンプを確認し、**前回取得から20分以内のアクセス時はスクレイピングをスキップしてデータベースのデータを表示する**仕組み（スロットリングガード）を取り入れています。
